## 分层设计

### 1. CLI 层 — `cli.py`

用户交互入口，基于 argparse 实现子命令路由。每个子命令对应一个 handler 函数，负责：
- 解析参数
- 组装依赖（config → pool → 核心模块）
- 格式化输出（table / json）

### 2. Core 核心模块 — `core/`

| 模块 | 文件 | 职责 |
|------|------|------|
| 节点对比 | `node_comparator.py` | 调度后端引擎，并行对比多个表，汇总结果 |
| 应用-DB对比 | `app_db_comparator.py` | 记录应用写入日志，回验数据库实际数据 |
| RPO检测 | `rpo_checker.py` | 故障前打标记/快照，故障后验证数据完整性 |

### 3. Backends 后端引擎 — `backends/`

可插拔的对比引擎，每个引擎实现相同的语义：输入两个节点+表名，输出差异报告。

```
                    ┌──────────────────┐
                    │  node_comparator  │
                    └────────┬─────────┘
                             │
            ┌────────────────┼────────────────┐
            │                │                │
       ┌────▼────┐     ┌────▼────┐     ┌─────▼─────┐
       │checksum │     │ native  │     │ 外部工具   │
       │(内置)   │     │ (内置)  │     │pg_comparator
       └─────────┘     └─────────┘     │data-diff   │
                                       │pgdiff      │
                                       └───────────┘
```

#### checksum 引擎原理

```
┌──────────────────────────────────────────────────┐
│  Table: accounts (100万行, PK: id)                │
│                                                   │
│  Chunk 0        Chunk 1        Chunk N            │
│  [id 1-10000]   [id 10001-20000]  ...             │
│  │              │              │                   │
│  ▼              ▼              ▼                   │
│  MD5(row1||     MD5(row10001   ...                │
│  row2||...)     ||row10002...)                    │
│  │              │              │                   │
│  ▼              ▼              ▼                   │
│  对比 Node A chunk checksum vs Node B chunk checksum│
│  不一致的chunk = 数据差异所在区间                     │
└──────────────────────────────────────────────────┘
```

- 将表按主键范围分成 `chunk_size` 大小的块
- 每块计算 `MD5(string_agg(MD5(row), ORDER BY pk))`，内层子查询同时输出行哈希和主键列
- 对比两个节点的块 checksum
- **优点**：速度快，适合大表；**缺点**：只知道哪块不一致，不知具体行

#### native 引擎原理

```sql
-- Node A 有但 Node B 没有的行
SELECT * FROM node_a.accounts
EXCEPT
SELECT * FROM node_b.accounts;

-- Node B 有但 Node A 没有的行
SELECT * FROM node_b.accounts
EXCEPT
SELECT * FROM node_a.accounts;
```

- **优点**：精确到行，能看到具体差异数据
- **缺点**：大表时较慢，需全表扫描

### 4. Workload 负载模块 — `workload/`

```
┌─────────────────────────────────────────────────┐
│              workload/generator.py               │
│                                                  │
│  DataGenerator: 测试数据生成器                     │
│  ├── create_all_tables()  建表（4张预定义表）      │
│  ├── populate_*()         按批次插入数据            │
│  ├── truncate_all()       清空表数据（CASCADE）    │
│  └── teardown()           删除所有表               │
│                                                  │
│  workload/runner.py                              │
│  YAML解析 → 建表 → 种子数据 → 多线程并发执行       │
│                                                  │
│  Worker-1  Worker-2  ...  Worker-N              │
│  │          │               │                    │
│  ▼          ▼               ▼                    │
│  ┌─────────────────────────────┐                 │
│  │  按权重随机选择事务SQL        │                 │
│  │  %(random_int) → 随机替换    │                 │
│  │  执行 → 记录统计              │                 │
│  └─────────────────────────────┘                 │
└─────────────────────────────────────────────────┘
```

YAML 模板支持的随机占位符：

| 占位符 | 示例 | 替换结果 |
|--------|------|---------|
| `%(random_int:N-M)s` | `%(random_int:1-100)s` | 1 到 100 的随机整数 |
| `%(random_str:N)s` | `%(random_str:16)s` | 16 位随机字符串 |
| `%(random_float:N-M)s` | `%(random_float:0.01-1000)s` | 随机浮点数 |
| `%(uuid)s` | `%(uuid)s` | UUID |
| `%(timestamp)s` | `%(timestamp)s` | 当前时间戳 |

### 5. Utils 工具层 — `utils/`

| 模块 | 职责 |
|------|------|
| `db.py` | 连接池管理，支持多节点连接复用；提供游标、主键查询、行数统计等辅助方法。`execute()` 自动判断单条/批量参数（`execute` vs `executemany`），写操作自动提交 |
| `config.py` | YAML 配置加载，合并默认值，校验必填字段。`generate_template()` 从 `config.yaml.example` 模板生成配置文件 |
| `logger.py` | 统一的日志格式，支持控制台+文件双输出 |

**Python 3.8 兼容性：** 所有使用 PEP 585 类型标注（如 `list[str]`）的模块均添加了 `from __future__ import annotations`，确保在 Python 3.8 上可正常运行。

## RPO 检测原理

RPO（Recovery Point Objective）即恢复点目标，衡量故障后可能丢失的数据量。

```
时间线:
                   故障发生点
                      │
  ────────────────────┼──────────────────────▶
   [正常写入]          │    [恢复后验证]
                      │
  ① plant_markers()   │  ② check_all()
     - 插入标记行          - 检查标记是否存在（标记丢失 = 数据丢失）
     - 记录行数快照        - 对比行数（行数减少 = 数据丢失）
     - 记录 MAX(id)       - 检查ID空洞（序列断档 = 数据丢失）
```

三种检测手段：

1. **标记检测**：在目标表中插入带唯一 ID 的记录，故障后检查是否还存在
2. **行数对比**：记录故障前 COUNT(*)，故障后对比，精确量化丢失行数
3. **序列空洞检测**：对自增主键列，通过 `generate_series` 找缺失 ID，区分"被删除"和"故障丢失"

## 应用-DB一致性原理

```
应用层                           数据库层
  │                                │
  │ INSERT account (id=1, bal=100) │
  │ ─────────────────────────────▶ │
  │                                │
  │ 记录到 _app_db_tracking:       │
  │   op=INSERT, pk={id:1},       │
  │   row_data={id:1,bal:100}     │
  │                                │
  │           ...业务运行中...       │
  │                                │
  │ verify_all():                  │
  │   SELECT * FROM accounts       │
  │   WHERE id=1 ────────────────▶ │
  │   ◀── 对比 row_data 与实际数据  │
  │                                │
  │   不一致 → 报告差异             │
  │   不存在 → 报告缺失             │
```
